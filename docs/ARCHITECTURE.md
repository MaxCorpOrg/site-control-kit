# Архитектура

## Компоненты

1. `webcontrol` (Python)
- HTTP API сервер (`webcontrol.server`).
- Хранилище состояния + очередь команд (`webcontrol.store`).
- CLI (`webcontrol.cli`) для оператора и скриптов.

2. Расширение браузера (`extension/`)
- `background.js`: heartbeat, polling, выбор вкладки, отправка результатов.
- `content.js`: выполнение DOM-команд в контексте страницы.
- `options/popup`: интерфейс настроек и диагностики.

3. Оператор / внешняя автоматизация
- Ручной запуск команд через `sitectl`.
- Любой другой инструмент может работать через тот же HTTP API.

## Поток данных

## A) Регистрация клиента
1. Расширение стартует.
2. Отправляет `POST /api/clients/heartbeat` с `client_id`, списком вкладок и метаданными.
3. Хаб обновляет запись клиента и его очередь.

## B) Жизненный цикл команды
1. CLI вызывает `POST /api/commands`.
2. Хаб валидирует payload и кладёт ID команды в очередь целевых клиентов.
3. Расширение запрашивает `GET /api/commands/next?client_id=...`.
4. Хаб отдаёт следующую команду, статус доставки становится `dispatched`.
5. Расширение выполняет действие и отправляет `POST /api/commands/{id}/result`.
6. Хаб сохраняет результат и пересчитывает агрегированный статус команды.

## C) Диагностика и мониторинг
- `GET /api/state` — полный снимок.
- `GET /api/clients` — видимые клиенты и вкладки.
- `GET /api/commands/{id}` — полный статус конкретной команды.

## Хранилище

Состояние хранится в одном JSON-файле (`state.json`):
- `clients` — подключённые клиенты.
- `queues` — FIFO-очереди по клиентам.
- `commands` — команда + статусы доставок + результаты.

## Статусы

Статусы доставки (на клиента):
- `pending`
- `dispatched`
- `completed`
- `failed`
- `cancelled`
- `expired`

Агрегированные статусы команды:
- `pending`
- `in_progress`
- `completed`
- `failed`
- `partial`
- `cancelled`
- `expired`
- `rejected`

## Маршрутизация

Цель команды может задаваться:
- `client_id` — один клиент.
- `client_ids` — несколько клиентов.
- `broadcast=true` — все клиенты.

Выбор вкладки внутри клиента:
1. `target.tab_id`
2. `target.url_pattern` (вхождение подстроки в URL)
3. активная вкладка (`target.active !== false`)
4. первая доступная вкладка

## Устойчивость
- MV3 service worker не постоянный и может выгружаться.
- Используются два механизма пробуждения/опроса:
  - `setInterval`
  - `chrome.alarms`
- Очередь и история команд сохраняются на стороне хаба.

## Совместимость и развитие
- Изменения протокола делать по принципу обратной совместимости.
- Для ломающих изменений:
  - обновлять документацию,
  - описывать миграцию,
  - по возможности добавлять fallback.
