# Telegram Client Roadmap

## Текущее состояние

Проект уже рабочий как локальный browser-control kit:
- хаб `webcontrol` принимает heartbeat и команды;
- браузерное расширение исполняет DOM- и tab-level команды;
- Telegram-экспорт из чата умеет собирать часть `@username`;
- батч-сценарий для новых контактов уже работает через `scripts/collect_new_telegram_contacts.sh`.

Сильные стороны:
- мало зависимостей;
- рабочий живой контур `hub -> extension -> DOM`;
- понятный CLI;
- документация и минимальные unit-тесты уже есть.

Слабые стороны:
- Telegram-экспорт слишком завязан на текущий DOM Telegram Web;
- deep-логика всё ещё может ошибочно сопоставить `@username` не тому `peer_id`;
- нет полноценного интеграционного покрытия живого браузерного контура;
- экспортный код монолитный и тяжёлый для сопровождения.

## Критичные доработки

### P0. Достоверность `@username`

Нужно:
- ввести `confidence` для источника username: `mention`, `url`, `profile`, `manual`;
- не писать в numbered batch сомнительные совпадения;
- складывать спорные случаи в `review.txt` или `conflicts.json`;
- вести историю `peer_id -> username` и `username -> peer_id` между запусками;
- запрещать смену владельца `@username` без явного подтверждения.

Почему это важно:
- сейчас главный риск не в том, что контакт не найден, а в том, что найден неверный контакт.

### P0. Явное логирование прогона

Нужно:
- отдельная папка лога на каждый запуск;
- сохранять `run.json` с параметрами запуска;
- хранить сырые временные `latest_full.md/txt`;
- сохранять ошибки DOM-поиска и проблемные `peer_id`.

Почему это важно:
- без следов не разобрать, где Telegram UI сломал проход.

### P1. Разделение экспортера на модули

Разбить `export_telegram_members_non_pii.py` на:
- `hub_client.py`
- `telegram_dom.py`
- `telegram_parser.py`
- `telegram_identity.py`
- `telegram_reports.py`

Почему это важно:
- сейчас один файл объединяет HTTP, DOM-логику, парсинг, дедупликацию и отчёт.

### P1. Выбор вкладки Telegram

Нужно:
- работать по конкретному `tab_id`, если чат выбран из списка вкладок;
- не полагаться только на `url_pattern`;
- помечать активную целевую вкладку в GUI.

Почему это важно:
- у пользователя часто открыто несколько одинаковых Telegram tabs.

### P1. Интеграционные тесты

Нужно:
- добавить browser smoke для `background.js` и `content.js`;
- проверять `click`, `click_text`, `scroll_by`, `get_html`, `wait_selector`;
- добавить fixture-HTML для Telegram DOM-вариантов;
- проверить edge-cases для duplicate `@username`.

Почему это важно:
- unit-тесты сейчас не покрывают реальный browser bridge.

## Полезные доработки

### P2. Очередь и состояние хаба

Нужно:
- pruning старых команд из `state.json`;
- лимит хранения истории;
- отдельный архивный файл или SQLite вместо бесконечного JSON;
- health-check и корректное завершение stale listeners.

### P2. Унификация скриптов

Сейчас есть несколько похожих сценариев:
- `telegram_members_export_app.sh`
- `telegram_members_export_gui.sh`
- `auto_collect_usernames.sh`
- `run_chat_export_once.sh`
- `collect_new_telegram_contacts.sh`

Нужно:
- выделить один основной CLI-flow;
- GUI строить поверх него;
- не дублировать параметры и дефолты по нескольким файлам.

### P2. Безопасный режим данных

Нужно:
- хранить batch-историю в SQLite;
- записывать номер батча, дату, URL чата, число новых username;
- легко отвечать на вопрос "когда этот username появился впервые?".

## Инструменты под нашу логику

### 1. Оставить `site-control-kit` как оркестратор

Подходит для:
- локального управления реальным браузером;
- живого обхода Telegram Web;
- ручной отладки расширения и DOM.

Роль:
- текущая основа;
- не заменять, а усиливать.

### 2. Playwright

Подходит для:
- интеграционных тестов браузерного контура;
- стабильных locator-based smoke-тестов;
- записи trace и разборов flaky-поведения.

Почему подходит:
- строгие локаторы, auto-waiting, trace viewer.

Где использовать:
- не как замену текущему runtime Telegram-экспорта;
- а как основной инструмент тестирования и отладки browser layer.

Ссылки:
- https://playwright.dev/python/docs/locators
- https://playwright.dev/python/docs/trace-viewer-intro

### 3. Pyrogram

Подходит для:
- прямой работы с Telegram через user session;
- получения участников чата без DOM-парсинга;
- гибридного режима: `API first`, браузер как fallback.

Почему подходит:
- `get_chat_members()` и `get_chat_member()` уже дают модель участников на уровне API.

Риск:
- ограничения прав доступа и политики Telegram всё равно нужно учитывать.

Ссылки:
- https://docs.pyrogram.org/api/methods/get_chat_members
- https://docs.pyrogram.org/api/methods/get_chat_member

### 4. Telethon

Подходит для:
- более низкоуровневой MTProto-работы;
- случаев, где нужен точный контроль над peer/ref и историей доступа;
- тонкой Telegram-логике, если Pyrogram окажется слишком абстрактным.

Почему подходит:
- гибкий доступ к Telegram peer/model layer.

Минус:
- выше сложность сопровождения.

Ссылка:
- https://docs.telethon.dev/en/v2/concepts/peers.html

### 5. TDLib

Подходит для:
- долгоживущего production-демона;
- большого объёма Telegram-операций;
- более “официального” клиентского слоя, чем DOM-автоматизация.

Почему подходит:
- если нужен следующий шаг после shell-скриптов и нестабильного UI.

Минус:
- самый тяжёлый по интеграции путь.

Ссылки:
- https://tdlib.github.io/td/
- https://core.telegram.org/api/channel

### 6. Telegram Bot API

Подходит ограниченно:
- для чат-ботов и сервисной логики;
- для админских операций, когда бот уже добавлен и имеет нужные права.

Не подходит как основа вашего кейса:
- не заменяет user-session сценарий выгрузки из произвольного чата;
- не даёт тот же контроль, что пользовательский клиент/API.

Ссылки:
- https://core.telegram.org/bots
- https://core.telegram.org/bots/api

### 7. Selenium

Подходит ограниченно:
- если нужен WebDriver-контур из корпоративной среды;
- если Playwright по каким-то причинам нельзя использовать.

Для этого проекта подходит хуже Playwright:
- больше ручных ожиданий;
- меньше удобства для стабильного smoke/debug.

Ссылка:
- https://www.selenium.dev/documentation/webdriver/support_features/expected_conditions/

## Рекомендуемая стратегия

### Ближайший этап

Оставить текущий стек и сделать:
- `safe mode` для Telegram username;
- `review/conflicts` слой;
- нормальные run-logs;
- интеграционные тесты browser bridge.

### Средний этап

Сделать гибрид:
- сначала пробовать Telegram API-клиент;
- browser export использовать как fallback;
- batch-файлы наполнять только подтверждёнными username.

### Дальний этап

Перейти от shell-сценариев к сервису:
- SQLite ledger;
- task runner;
- стабильный Telegram client layer;
- GUI поверх нормального backend workflow.

## Практический вывод

Если выбирать один следующий инструмент под текущую логику, лучший порядок такой:
1. Playwright для тестирования и дебага browser-контура.
2. Pyrogram как первый кандидат на замену хрупкого DOM-сбора.
3. Telethon или TDLib, если нужна более глубокая и надёжная Telegram-интеграция.

Если выбирать один следующий технический шаг в проекте:
1. внедрить `safe mode` и quarantine для сомнительных username;
2. после этого начинать гибрид с Telegram API.
