# Гайд Для ИИ-Сопровождения

Документ для агентов, которые используют, поддерживают и улучшают `site-control-kit`.

## Что Это За Инструмент
`site-control-kit` — локальный контур управления браузером.
Он нужен, чтобы агент мог:
- видеть подключённые браузеры и вкладки;
- открывать страницы и вкладки;
- кликать, вводить, ждать селекторы;
- читать текст, HTML, URL, атрибуты;
- делать скриншоты;
- выполнять команды в реальном браузере через расширение.

Проект не является абстрактной библиотекой. Это прикладной рабочий инструмент для живого браузерного управления.

## Основной Принцип
Если репозиторий доступен, сначала использовать `site-control-kit`, а не придумывать обходной путь.

На Windows стандартный вход:
- `start-hub.cmd`
- `browser.cmd`

В Python-окружении:
- `python -m webcontrol ...`
- `sitectl ...`
- `sitectl browser ...`

## Из Чего Состоит Система

### 1. Хаб
Файлы:
- `webcontrol/server.py`
- `webcontrol/store.py`

Роль:
- принимает heartbeat от клиентов;
- хранит состояние и очередь;
- маршрутизирует команды;
- принимает результаты выполнения.

### 2. CLI
Файл:
- `webcontrol/cli.py`

Роль:
- даёт оператору и агенту короткий интерфейс;
- скрывает низкоуровневый API;
- реализует `sitectl browser` как удобный слой поверх протокола.

### 3. Расширение
Файлы:
- `extension/background.js`
- `extension/content.js`

Роль:
- `background.js` работает с вкладками, heartbeat, polling, скриншотами;
- `content.js` работает с DOM внутри страницы.

### 4. Обёртки Для Windows
Файлы:
- `start-hub.cmd`
- `start-hub.ps1`
- `browser.cmd`
- `browser.ps1`
- `scripts/*.cmd`
- `scripts/*.ps1`

Роль:
- быстрый запуск без длинных команд;
- удобство для пользователя и агента в реальной Windows-среде.

## Источники Правды
- `docs/PROJECT_WORKFLOW_RU.md` — обязательный порядок работы, проверок и handoff.
- `docs/PROJECT_STATUS_RU.md` — текущее состояние проекта, завершённые задачи, открытые проблемы и следующий приоритет.
- `docs/API.md` — типы команд, поля payload, контракт результата.
- `docs/ARCHITECTURE.md` — поток команд и распределение ролей.
- `docs/EXTENSION.md` — где реализованы browser-level и DOM-level действия.
- `AGENTS.md` — политика поведения агентов внутри репозитория.
- `BROWSER_QUICKSTART.md` — кратчайший практический маршрут.

## Рекомендуемый Порядок Работы Агента
1. Прочитать `AGENTS.md`.
2. Прочитать `docs/PROJECT_WORKFLOW_RU.md`.
3. Прочитать `docs/PROJECT_STATUS_RU.md`.
4. Прочитать `BROWSER_QUICKSTART.md`.
5. Прочитать `docs/API.md`.
6. Проверить живой контур:
   - `browser.cmd status`
   - `browser.cmd tabs`
7. Выполнить нужные действия через `browser.cmd` или `sitectl browser`.
8. Если возможностей не хватает, расширить инструмент, а не плодить временные костыли.

## Базовый Smoke-Тест
Минимальный живой сценарий:

```cmd
start-hub.cmd
browser.cmd status
browser.cmd tabs
browser.cmd open https://example.com
browser.cmd text h1
```

Ожидаемое поведение:
- в `status` виден подключённый клиент;
- в `tabs` видны вкладки клиента;
- страница открывается;
- `text h1` возвращает содержимое заголовка.

## Как Добавлять Новые Возможности

### Если это новая DOM-команда
Примеры:
- новый способ клика;
- чтение нового типа данных со страницы;
- работа с формами;
- ожидание или навигация внутри страницы.

Менять:
- `extension/content.js`
- при необходимости `webcontrol/cli.py`
- `docs/API.md`
- `docs/EXTENSION.md`
- `examples/`

### Если это tab-level или browser-level команда
Примеры:
- новая операция с вкладкой;
- screenshot;
- reload/activate/close tab;
- работа с несколькими окнами.

Менять:
- `extension/background.js`
- при необходимости `webcontrol/cli.py`
- `docs/API.md`
- `docs/EXTENSION.md`
- `examples/`

### Если это удобство использования
Примеры:
- более короткая команда в CLI;
- новый alias;
- улучшение выбора клиента;
- Windows-обёртка;
- человекочитаемая ошибка.

Менять:
- `webcontrol/cli.py`
- `scripts/*.cmd` или `scripts/*.ps1`
- `README.md`
- `BROWSER_QUICKSTART.md`
- при изменении поведения `AGENTS.md`

## Инварианты, Которые Нельзя Ломать
- Хаб должен оставаться единым источником правды.
- Очередь клиента должна быть детерминированной.
- Агрегированный статус команды должен соответствовать реальным доставкам.
- Формат ошибок должен оставаться понятным и диагностируемым.
- Автовыбор клиента и вкладки не должен становиться неожиданным.

## Типовые Ограничения
- `chrome://*` и аналогичные страницы не управляются content script.
- Некоторые сайты блокируют `run_script` через CSP.
- MV3 service worker может засыпать, поэтому нельзя убирать резервный polling без замены.
- На защищённых сайтах часть действий может требовать работу через DOM-команды вместо JS.

## Политика Улучшений
Приоритет улучшений:
1. надёжность;
2. короткие команды;
3. понятные ошибки;
4. Windows-friendly запуск;
5. синхронность документации и кода.

Хорошие изменения:
- меньше ручных аргументов для типовых задач;
- явные команды вместо `send --type ...`;
- более безопасный выбор клиента;
- лучшее логирование ошибок;
- примеры, которые реально можно запускать.

Плохие изменения:
- немые изменения контракта;
- новые возможности без документации;
- новые возможности без smoke-проверки;
- улучшения, которые усложняют обычный повседневный сценарий.

## Что Делать Перед Коммитом
1. Запустить:
   - `PYTHONPATH="$PWD" python3 -m unittest discover -s tests -p 'test_*.py'`
2. Если менялся CLI:
   - `PYTHONPATH="$PWD" python3 -m webcontrol --help`
   - `PYTHONPATH="$PWD" python3 -m webcontrol browser --help`
3. Если менялся браузерный контур:
   - `browser.cmd status`
   - `browser.cmd tabs`
   - минимум один живой сценарий в браузере
4. Обновить `docs/PROJECT_STATUS_RU.md`, чтобы новый агент видел:
   - что завершено;
   - что проверено;
   - что остаётся проблемой;
   - какой следующий приоритет.
5. Обновить документацию так, чтобы новый агент мог начать работу без устных пояснений.
