# Гайд для ИИ-сопровождения

Документ для ИИ-агентов, которые поддерживают и развивают проект.

## Цели
- Сохранять стабильность для пользователя.
- Не ломать API-контракт без явной миграции.
- Делать изменения наблюдаемыми и документированными.

## Контрактные границы
1. API хаба — главный контракт.
2. Расширение — исполнитель команд, а не источник состояния.
3. `state.json` должен оставаться читаемым и диагностируемым вручную.

## Рабочий процесс изменений
1. Прочитать `docs/API.md` и `docs/ARCHITECTURE.md`.
2. При изменении протокола:
- обновить документацию,
- обновить примеры,
- по возможности сохранить обратную совместимость.
3. Запустить тесты и smoke-проверки.
4. Обновить README при изменении UX/CLI.

## Ответственность по файлам
- `webcontrol/store.py` — очередь, статусы, persistence.
- `webcontrol/server.py` — auth, маршрутизация, HTTP-контракт.
- `webcontrol/cli.py` — пользовательский CLI и ошибки.
- `extension/background.js` — heartbeat, polling, orchestration.
- `extension/content.js` — DOM-операции.

## Инварианты
- Каждая команда имеет стабильный `id`.
- Агрегированный статус соответствует статусов доставок.
- Клиент должен отправлять результат по полученной команде.
- Неавторизованные запросы не должны раскрывать внутреннее состояние.

## Рекомендуемый roadmap
1. WebSocket push-режим вместо постоянного polling.
2. Пер-клиентные токены.
3. Версионируемая схема команд (JSON Schema).
4. Поддержка frame-targeting.
5. Серверное сохранение screenshot-артефактов.
6. Опциональное шифрование state-хранилища.

## Минимальный smoke-тест
1. Запустить хаб.
2. Проверить, что клиент появился в `clients`.
3. Отправить `navigate`.
4. Отправить `wait_selector` или `extract_text`.
5. Проверить команду через `/api/commands/{id}`.

## Политика обработки ошибок
- Ошибки должны быть явными и пригодными для диагностики.
- Не глушить ошибки silently.
- Добавлять понятные `hint`, если проблема типовая.
